# Minimum Foundation Platform (MFP)

## Purpose

The Minimum Foundation Platform (MFP) defines the smallest implementation of VentureOS capable of demonstrating its core philosophy.

The purpose of the MFP is not to build a complete venture operating system.

Its purpose is to prove that an evidence-led, object-centric, governed approach produces better decisions than a traditional AI chat workflow.

Every feature included in the MFP must directly support that objective.

Everything else is deferred.

---

# Success Definition

The MFP is successful if a user can begin with an empty Case and complete one full VentureOS Operational Lifecycle.

Starting with no knowledge, the system should support:

Observation

↓

Evidence

↓

Hypothesis

↓

Challenge

↓

Decision

↓

Action

↓

Outcome (manual if necessary)

↓

Lesson

↓

Updated organisational knowledge

The lifecycle must be fully traceable.

---

# Core Demonstration

The MFP should allow a user to investigate a single business opportunity from initial idea through to an evidence-backed Board decision.

Everything required to justify that decision should remain permanently recorded.

---

# Primary User

The initial user is the Founding Board.

No multi-user capability is required.

Authentication may be minimal.

The focus is organisational capability rather than user management.

---

# Essential Capabilities

## Case Management

The user can:

* create a Case
* edit Case details
* archive a Case
* reopen a Case
* browse Cases

---

## Observation Management

The user can:

* create observations
* import observations from documents
* import observations from websites
* tag observations
* link observations to Cases

---

## Evidence Management

The user can:

* create evidence from observations
* link evidence to hypotheses
* assign confidence
* record supporting sources
* record contradictory evidence

---

## Hypothesis Management

The user can:

* create hypotheses
* create assumptions
* create opportunities
* link evidence
* revise hypotheses
* change status

---

## Red Team Review

The system can:

* analyse hypotheses
* identify missing evidence
* identify weak assumptions
* identify contradictions
* raise structured objections
* suggest mitigation

The objective is constructive challenge.

---

## Board Review

The system can prepare a Board briefing containing:

* executive summary
* confidence
* evidence
* objections
* counterarguments
* risks
* recommended next action

The human remains responsible for approval.

---

## Decision Recording

The user can:

* approve
* reject
* defer
* request further research

Every decision is permanently recorded.

---

## Timeline

Every significant event automatically appears in the Case timeline.

Examples include:

* observation added
* evidence created
* hypothesis proposed
* objection raised
* decision made
* lesson created

The timeline forms the audit trail.

---

## Lessons

The user can record lessons learned.

Lessons remain linked to their originating Case.

Future Cases should be able to reference previous lessons.

---

# Essential Objects

The MFP requires only the following objects:

* Case
* Observation
* Evidence
* Hypothesis
* Assumption
* Opportunity
* Objection
* Counterargument
* Decision
* Lesson

Other objects are intentionally deferred.

---

# Essential Roles

Only three organisational roles are required.

## Research

Creates observations and evidence.

---

## Red Team

Challenges reasoning.

Raises objections.

Requests additional evidence.

---

## Board

Reviews evidence.

Makes decisions.

Records rationale.

---

# Artificial Intelligence

The MFP should use AI only where it adds measurable value.

Examples include:

* summarisation
* classification
* extracting observations
* Red Team reasoning
* Board briefing preparation

AI should not replace governance.

---

# Local First

The MFP should operate entirely using local resources where practical.

Preferred principles:

* local LLMs
* local embeddings
* local vector search
* local database
* open-source tooling

Cloud services remain optional.

---

# Non-Goals

The MFP intentionally excludes:

* autonomous coding
* autonomous deployment
* marketing automation
* finance department
* CRM
* customer support
* self-improvement
* multi-user collaboration
* advanced permissions
* scheduling
* project management
* accounting
* sales pipelines
* agent marketplaces
* plugin ecosystems

These may be considered after the philosophy has been validated.

---

# Definition of Done

The MFP is complete when the following demonstration is possible.

A user creates a new Case.

The system gathers observations from local documents and web research.

Observations are converted into evidence.

A hypothesis is proposed.

The Red Team raises structured objections.

Additional evidence is collected.

The Board receives a briefing.

A decision is recorded.

The complete reasoning process is permanently preserved.

The user can return weeks later and fully understand:

* what was known
* what was uncertain
* what objections existed
* why the decision was made
* what happened afterwards

---

# Success Metrics

The MFP succeeds if it demonstrates:

* complete traceability
* evidence-led reasoning
* structured challenge
* transparent decision making
* persistent organisational knowledge
* repeatable workflow

Success is not measured by the number of AI agents.

Success is measured by the quality of organisational reasoning.

---

# Guiding Principle

Whenever implementation choices become difficult, choose the option that best supports the mission.

> Turn uncertainty into evidence.

> Turn evidence into decisions.

> Turn decisions into successful ventures.

---

# Implementation Impact

This document defines the implementation boundary for Version 0.1.

Any feature not required to demonstrate the VentureOS philosophy should be deferred until after the Minimum Foundation Platform has been successfully completed.

The objective is not to build everything.

The objective is to build enough to prove that VentureOS works.
